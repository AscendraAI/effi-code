# super-agent

**비용을 최소화한 멀티모델 코딩 에이전트 오케스트레이션.**
클로드를 대장으로 두고, 돈은 가장 적게 쓰고, 유료 사용량이 바닥나면 내 컴퓨터 안의 무료 모델이 이어받게 합니다.

> 거창한 프레임워크가 아닙니다. 새로 깔 서버도, 백그라운드에서 도는 프로그램도 없습니다.
> 그냥 **파일 몇 개와 "일하는 방식"**입니다. 모든 판단·승인·기록이 파일로 남습니다.

**성능이 필요한 곳엔 최고 모델, 나머지는 싼 모델, 사용량이 바닥나면 로컬로.** — 이게 철학의 전부입니다.

---

## super-agent이 뭔가요?

super-agent은 하나의 코딩 세션(Claude Code)을 **작은 팀**처럼 굴러가게 만듭니다. 실력 좋은 **클로드** 대장, 거드는 **코덱스·제미나이**, 그리고 내 컴퓨터에서 공짜로 일하는 **로컬 모델**. 팀장(오케스트레이터)이 일마다 "가장 싸게 해낼 수 있는 티어"로 보내고, 꼭 필요할 때만 위로 올립니다.

두 가지 고민을 풉니다.

- **비용** — 대부분은 크든 작든 모든 일을 제일 비싼 모델에게 시킵니다. 그런데 실제 코딩 토큰의 약 85%는 최상위 모델이 필요 없습니다. 쉬운 일은 아래로 내리고, 어려운 판단만 클로드에게.
- **한도** — 유료 요금제엔 사용량 한도가 있습니다. 한창 몰입하다 막히는 순간, 로컬 모델이 이어받아 작업이 끊기지 않게 합니다.

## 핵심 아이디어: 싼 티어부터

```
로컬 (공짜)  →  코덱스  →  제미나이  →  클로드 (최상위)
        싸게 먼저 시도하고, 결과가 부족할 때만 위로 올린다
```

| 티어 | 역할 | 비용 |
|---|---|---|
| **클로드** | 어려운 설계·디버깅·아키텍처·오케스트레이션 | 높음 → 아껴 씀 |
| **코덱스 / 제미나이** | 구현 보조·긴 문서·제3자 검토 | 낮음 |
| **로컬** (Ollama: Qwen / Devstral) | 좁고 기계적인 코딩·한도 백스톱 | 공짜 (느림) |

**메모리는 곧 파일 시스템입니다.** 런타임 상태가 없습니다 — 모든 작업·승인·판정이 파일로 남죠. 세션이 꺼져도 폴더 하나만 읽으면 이어서 할 수 있습니다.

## 시작하기

### 준비물
- macOS (Apple Silicon)
- [Claude Code](https://claude.com/claude-code) — 오케스트레이션 하네스
- [Ollama](https://ollama.com) — 로컬 모델 구동 (`setup.sh`가 대신 설치)

### 설치
```bash
git clone https://github.com/AscendraAI/super-agent && cd super-agent

./setup.sh                                    # Ollama 설치 + 내 맥에 맞는 로컬 모델 자동 선택·다운
ln -s "$PWD/bin/sag"      /opt/homebrew/bin/sag
ln -s "$PWD/bin/sag-run"  /opt/homebrew/bin/sag-run
ln -s "$PWD/bin/sag-pick" /opt/homebrew/bin/sag-pick
```

### 설정
- `ORCHESTRATION.md`를 세션 규칙으로 물려 둡니다(예: 프로젝트의 `CLAUDE.md`). 오케스트레이터가 매 작업 시작 시 읽습니다.
- 로컬 모델은 **가용 메모리에 맞춰 자동 선택**되니 따로 설정할 필요가 없습니다. 고정하려면 `SA_LOCAL_MODEL=devstral sag local`.

## 사용법

```bash
sag            # 평소 = 구독 클로드 (제일 똑똑하게)
sag local      # 사용량이 바닥나면 → 무료 로컬로 (남은 메모리에 맞춰 자동 선택)
sag status     # 지금 어떤 로컬 모델이 적당한지 확인
```

지루하고 양 많은 일은 로컬에 바로 위임할 수 있습니다.

```bash
sag-run "다음 5개 UI 문자열을 자연스러운 한국어로: Save, Cancel, Delete, Loading, Done"
```

그리고 평소 클로드 세션에서는, 오케스트레이터가 지루한 대량 작업을 **먼저 제안**합니다.

> "이건 기계적인 대량 작업이라 무료 로컬로 돌려서 사용량을 아낄 수 있어요. 그렇게 할까요?"

## 왜 이렇게 만들었나

모든 선택은 2026년 최신 연구와 실제 측정을 근거로 합니다. 요약하면 다섯 가지죠.

1. **클로드가 대장을 맡되, 팀을 함부로 늘리지 않는다.** 강한 대장 + 싼 팀원이 대장 혼자보다 낫다는 건 실측(Anthropic +90%)이 있습니다 — 다만 그건 *조사·연구* 일입니다. *코딩*에선 팀을 여럿 굴리면 토큰을 15배 쓰면서도 손발이 안 맞아 오히려 나빠질 수 있습니다. 그래서 강한 클로드 한 스레드로, 서브에이전트는 정말 병렬 가능한 일에만.
2. **약하게 만들지 말고, 라우팅으로 아낀다.** 토큰의 85%는 최상위가 필요 없습니다. 캐싱은 최대 90%까지 아끼지만, 대화를 여러 회사 모델로 쪼개면 캐시가 깨져 오히려 10배가 됩니다. 그래서 대장의 대화는 한 곳에서 쭉.
3. **로컬은 대체가 아니라 백스톱이다.** 일상 작업 ~80%를 커버하되, 느리고 덜 똑똑합니다. 구독을 프록시로 자동 전환하는 건 Anthropic 약관 위반이라, 전환은 정직한 토글입니다.
4. **지루한 대량 작업은 로컬에 넘길지 먼저 물어본다.** 몰래 낮추지 않고 꼭 확인을 받습니다.
5. **로컬 모델은 가용 메모리에 맞춰 자동 선택**해 컴퓨터가 느려지지 않게 합니다.

자세한 근거와 출처: [`docs/why.md`](docs/why.md).

## 솔직한 한계

- 로컬은 여러 파일에 걸친 복잡한 작업에서 클라우드를 못 따라갑니다 — 어디까지나 백스톱입니다.
- 구독 사용자는 로컬로 **직접**(`sag local`) 전환해야 합니다 — 약관을 지키는 길이라서요.
- 앱을 많이 띄운 16GB 맥에서는 작은 로컬 모델까지만 쾌적합니다. 한계는 도구가 아니라 메모리의 몫입니다.
- 라우터·로컬 도구는 월 단위로 바뀝니다 — 설정 전에 각 공식 문서로 최신을 한 번 확인하세요.
